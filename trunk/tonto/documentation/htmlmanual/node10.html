Motivation for the critical requirements

The reason for having the module name introduced as a prestring for the module routine is to prevent namespace pollution. For example, the routine get_item may be a common possibility for a routine name; using the module name as a prefix prevents any possible name clash, if a list of unique module names is maintained. Similarly, the provision of a generic interface with an underscore as the first character (e.g. _get_item) also prevents namespace pollution with any local variables with the same name. In addition, the provision of a generic interface means that other modules may deliberately re-use the name of the routine without a name conflict. This possibility of overloading the same generic name by different modules should be employed whenever the overloaded routine name performs a similar function in the different modules.

Inheritance. Suppose MAT and CMAT are two modules which describe matrices of real and complex numbers, respectively. The same generic interface name _product might be used in both modules to describe a routine which returns the product of two matrices, whether real or complex. If every procedure in MAT had a corresponding procedure in CMAT with the same generic name, any code written initially using real matrices is trivially extended to complex matrices by changing the string MAT to CMAT in the program code.
